## 합의 알고리즘 비교

이전에 블록 및 트랜잭션 생성, 작업 증명과 지갑 생성 및 조회를 구현해 보고 난 후, 평소 개념적으로만 알고 있었던 합의 알고리즘의 내부 동작 과정을 접할 수 있었다.

이더리움은 2.0이 되면서 PoW 합의 방식에서의 **높은 가스비와 TPS**를 개선하기 위해 **PoS와 샤딩**을 도입했다. 이전에 구현했던 작업 증명과 이더리움을 통해 PoW와 PoS를 비교해보고자 한다.

## PoW(작업 증명)와 PoS(지분 증명)

지분 증명의 내부 동작 과정을 살펴보기에 앞서 작업 증명과 지분 증명을 간단한 정리 및 개념적 비교 후 넘어가려 한다.

> 두 방식의 주된 차이점은 블록 생성자 선정 방법과 Sybil 공격을 막는 방법에 대한 차이와 블록이 확정되는 과정 등이 있다.

#### PoW(작업 증명)

참여자들이 많은 계산을 필요로 하는 문제를 경쟁적으로 풀게 하고 보상을 지급하여 **중개자 없이** 네트워크의 신뢰성과 무결성을 유지한다. 컴퓨팅 자원과 리소스를 할애하여 특정 범위 내의 유효 해시를 찾는 **마이닝 작업**을 통해 네트워크에 기여한다.

유효한 체인인지 확인하는 규칙은 가장 긴 체인 규칙(Longest Chain Rule)을 따른다. 그래서 블록이 쌓일수록 더욱 긴 체인이 되고 유효한 체인으로 확정(확률론적 확정)될 확률이 높아진다. 작업 증명을 채택한 블록체인을 공격하기 위해선 이론상 그 블록체인의 **해시파워** 51%를 장악해야 한다.

#### PoS(지분 증명)

참여자들이 가상화폐 일정량을 담보로 스테이킹하면 그 중 랜덤하게 검증자를 선정한다. 담보 지분이 클수록 블록 제안자가 될 수 있는 확률이 높아진다. 일정 금액의 **가상화폐를 스테이킹** 하도록 요구해 **지분 기반**(**경제적**)으로 보안을 유지할 수 있게 되어 네트워크의 신뢰성을 보장할 수 있게 된다.

블록을 확정할 때 2/3 서명이 모이면 확정(결정론적 확정)된다. 이더리움의 경우 네트워크를 공격하기 위해서 스테이킹 된 **가상화폐** 2/3를 장악해야한다

#### 비교

1. 작업 증명은 네트워크가 확장되면서 마이닝에 하드웨어와 많은 **에너지**가 필요하다. 그러나 지분 증명에서 합의에 필요한 자원 형태는 전기나 연산력이 아닌 **재화**(지분)이기 때문에 네트워크 확장이 쉽다.
   -> **확장성** 면에서 지분 증명이 더 편리.

2. 작업 증명은 해시 계산 과정에서 에너지 소비가 많지만, 지분 증명은 스테이킹 된 노드가 블록을 생성하기 때문에 경쟁이 필요 없어 에너지 낭비가 적다
   -> **에너지 효율성** 면에서 지분 증명이 더 높음.

3. 작업 증명은 원칙적으로 누구나 하나의 하드웨어로 참여 가능. 그러나 지분 증명은 보유 지분에 따라 블록 제안 확률이 비례한다.
   -> **탈중앙화 정도**에서 이론상 작업 증명이 더 개방적, 지분 증명이 진입 장벽은 낮지만 실질적인 자본 집중으로 이어짐.

4. 작업 증명은 여러 블록이 쌓일수록 확정성(finality)이 증가하고, 지분 증명은 결정론적으로 블록이 확정된다
   -> **블록 확정**에 대한 차이점

5. 작업 증명 공격 비용은 **하드웨어**와 **에너지**이다. 지분 증명의 공격 자원은 스테이킹 지분으로 **현실 자원** -> 경제 자원을 통해 보안을 유지한다.

## 작업 증명 내부 동작 과정

작업 증명은 해시를 찾는 노드에게 블록 제안권이 주어지는 단순하고 직관적인 구조이다. 노드는 블록 헤더의 nonce를 바꿔가며 타겟보다 작은 해시를 찾는다. 난이도를 조정할 수 있으며 난이도를 통해 타겟 값이 낮아지면 유효 해시 범위가 작아져 작업 시간이 늘어나게 된다.

### PoW 나카모토 합의

PoW는 합의 알고리즘의 일부분이다. 완성된 합의 프로토콜은 다음과 같은 요소가 있다

1. PoW: Sybil 저항 + 블록 제안 권한 부여를 위함
2. Longest-chain rule: 체인 선택 규칙 - 어떤 체인이 유효한 지 선택하는 규칙으로 **가장 긴 체인**이 유효한 체인이 된다.
3. Probabilistic finality: 확률적 최종성 - 여러 블록이 쌓일수록 유효한 체인으로 굳어진다.

이러한 요소가 모인 합의 프로토콜이 **나카모토 합의**이다

### 유효 범위 해시값 마이닝 과정

이전에 해시를 찾는 과정을 구현한 것을 간단하게 정리했다

{/* prettier-ignore */}
<Callout type="info">
  새 블록 인스턴스가 생성되면 해당 블록 해시를 위해 마이닝 작업을 한다. (해시에는 블록 정보와 머클 트리 루트 해시가 들어간다)
</Callout>

마이닝은 특정 범위 내의 해시값을 찾는 작업을 의미한다. 타겟이라는 기준선을 설정한 후 기준보다 작은 범위의 해시를 찾는다.
정수인 난이도를 통해 타겟을 조절해서 이에 따라 유효 해시 범위가 달라진다.

**_타겟 설정_**

`t := new(big.Int).SetInt64()`
`t.Lsh(t, uint(256 - s.difficulty))`

{/* prettier-ignore */}
<Callout type="info">
  1 &lt;&lt; (256-난이도) 연산을 통해 해시 범위 설정. 난이도가 높을 수록 시프트 연산 수가 줄어들기 때문에 범위가 줄어든다
</Callout>

난이도 16, 17 비교

- 16: 1 &lt;&lt; 240 (=2^240), 상위 16비트
  --> 0000 0000 0000 0000 **1000** ... (총 256)
- 17: 1 &lt;&lt; 239 (=2^239), 상위 17비트
  --> 0000 0000 0000 0000 **0100** ... (총 256)
  결론적으로 난이도 16이 17보다 **범위가 넓어 큰 값을 허용**할 수 있다.

```go showLineNumbers title="pow.go"

type PowWork struct {
   Block *types.Block `json:"block"`
   Target *big.Int `json:"target"`
   Difficulty int64 `json:"difficulty"`
}

func (p *PowWork) RunMining() (int64, string) {
   var iHash big.Int
   var hash [32]byte

   merkleRoot := HashTransactions(p.Block)
   nonce := 0

   for nonce < math.MaxInt64 {
      d := getBytesForHash(merkleRoot, nonce)
      hash := sha256.Sum256(d)

      fmt.Printf("\r%x", hash)

      iHash.SetBytes(hash[:])

      if iHash.Cmp(p.Target) <= 0 {
         break
      } else {
         nonce++
      }
   }

   return Int64(nonce), hexutil.Encode(hash[:])
}


```

## 지분 증명 내부 동작 과정

지분 증명은 이더리움을 참고해 내부 동작 과정을 정리했다.

이더리움은 2.0이 되면서 기존 단일 실행 클라이언트 구조에서 실행 클라이언트 + 합의 클라이언트 형식으로 바뀌었다.

단일 클라이언트일 때는 확률론적인 확정성을 가졌었다. 가장 긴 체인이 진실이라는 단순하고 직관적인 규칙으로 합의가 성립되며 시간의 흔적을 남겨 신뢰를 축적했다.
2.0에서는 합의 클라이언트가 도입되어 결정론적인 확정성을 갖게 되었다. 투표 기반의 Gasper를 통해 기존 확률론적 확정성에서 바로 블록을 확정 지을 수 있게 됐다.

{/* prettier-ignore */}
<Callout type="info">

</Callout>

### 이더리움 PoS의 Gasper

Gasper은 **Casper FFG와 LMD GHOST**를 합친 이더리움 PoS의 핵심 프로토콜이다. Casper FFG와 LMD GHOST의 간략한 설명과 Gasper의 진행 흐름을 통해 정리했다.

**Casper FFG**
일정 간격마다 체크포인트를 생성하고 갱신한다. 체크포인트가 갱신되면서 이전 체크포인트와 **Supermajority Link**로 연결되며 이전으로 롤백할 수 없게 한다. **검증자의 2/3 이상**의 투표를 받아야 한다.

체크포인트가 확정되는 것은 참여자 간 동의를 얻어 네트워크를 보호하는 작업이다. 공격자가 체인을 이전으로 되돌리려고 해도 확정된 체크포인트 **이전으로 돌릴 수 없게** 된다.

**LMD GHOST**
**슬롯(Slot)**&#8203;마다 네트워크 지연 등 물리적인 요인으로 인해 필연적으로 **포크**가 발생한다. 포크가 일어날 경우 여러 포크에서 생성된 블록 중 어떤 것을 받아들일지에 대한 규칙 즉, **Fork Choice Rule**이다.
LMD GHOST는 "Latest Message Driven Greediest Heaviest Observed SubTree"의 약자로 최근 메시지 기반으로 가장 무거운 서브 트리를 따라간다는 의미이다.

위에서 나온 메시지는 **"증언 메시지"**&#8203;를 뜻한다. 이더리움 PoS에서 검증자는 블록 제안자와 증언자가 있다. 증언자는 어느 블록을 지지하는지 **증언 메시지(Attestation)**&#8203;를 통해 투표한다.
LMD GHOST는 포크 중 **가장 많은 최신의 증언 메시지**가 지지한 블록을 헤드로 정하게 된다.

Casper FFG에서는 **확정(finalize)**&#8203;에 대한 투표를 하고 LMD GHOST에서는 어느 포크의 리프 블록을 따라갈 것인지&#8203;**(Fork Choice Rule)**&#8203;에 대한 이야기이다. 자세한 내용은 Gasper 진행 흐름에서 정리한다.

### Gasper의 진행 흐름

이더리움 2.0의 Gasper 프로토콜의 흐름은 **슬롯(Slot)과 에폭(Epoch)**&#8203;이라는 시간 단위 기준으로 진행된다. 하나의 슬롯은 **12초** 단위로 하나의 블록이 생성되고 에폭은 **32슬롯**의 묶음 단위이다.

다음 슬롯에서 블록 제안자(proposer)는 새 블록을 제안할 때 여러 포크 중 **어떤 포크 체인에 새 블록**을 생성할 것인지 LMD GHOST에 의해 결정된다. 그리고 Casper FFG는 다음 에폭으로 넘어갈 때 현재 에폭의 블록들을 확정(finalize)하고 에폭 경계 블록 간 Supermajority Link를 연결할지 결정한다. 즉 LMD GHOST는 **슬롯 간 룰**이고 Casper FFG는 **에폭 간 투표**이다.

슬롯마다 검증자 그룹을 형성해 블록 생성 및 검증을 진행하는 **위원회(Committee)**&#8203;를 배정한다. 하나의 에폭이 시작될 때 전체 검증자를 32개의 위원회로 나누고 랜덤으로 배정한다.
블록 제안자와 증언자(검증자) 모두 LMD GHOST를 기반으로 가장 지지가 많은 HEAD를 찾아 증언 메시지에 포함한다.

![Gasper 진행 흐름](/contents/posts/blockchain/pos-consensus-with-ethereum/gasper-process.png)

**_슬롯(Slot)의 진행 흐름_**

1. 블록 제안자(proposer): LMD GHOST를 통해 제안할 블록을 연결할 포크를 찾아 블록 제안 시도
2. 위원회 검증인: 검증인들은 해당 블록에 대한 지지 여부를 증언 메시지를 통해 전파한다.
3. 이후 Attestation 데이터를 집계해서, LMD GHOST에 의해 다음 블록이 연결될 체인 HEAD를 결정할 때 사용

```go showLineNumbers title="증언 메시지(Attestation) 구조"
   // 증언 메시지에는 LMD GHOST와 Casper FFG 투표를 모두 포함한다
   type struct Attestation {
      Slot // 슬롯 번호
      CommitteeIndex // 위원회 번호
      Beacon_block_root // LMD GHOST 투표 포함

      // Casper FFG 투표(이미 정당화(justified)된 체크포인트 쌍)
      Soruce // 이전 에폭 경계 블록
      Target // 현재 에폭 경계 블록
   }

```

**_에폭(Epoch)의 진행 흐름_**

1. Epoch j와 j+1 경계 블록(첫번째 블록)은 이미 투표를 통해 정당화(justified) 되어 있다.
2. Epoch j+2부터 블록을 생성하며 LMD GHOST로 다음 블록을 연결할 포크를 계산하고 Casper FFG로 증언자의 Epoch j+1, j+2의 경계 블록 간 Supermajority Link에 대한 투표가 진행된다.
3. Casper FFG 투표로 지분 2/3 이상 투표가 진행되면 Epoch j+1과 j+2의 경계 블록 간 Supermajority Link로 연결된다.
4. Epoch j+1이 정당화된(justified) 체크포인트이기 때문에 이 링크와 연결된 Epoch j+2도 정당화되고, Epoch j+1에 해당하는 블록은 확정(finalize)된다.
